# P-4.1 — Adaptive Tick-Rate

**Fase:** [[DEVELOPMENT_MEMORY#Fase 4|Fase 4]]
**Depende de:** [[P-4.2 Thread Pool Model]] (necesita el profiler para medir)

## Problema

El roadmap asume 100Hz constante. Si el tick budget (10ms) se supera, el servidor acumula deuda y el tick-rate efectivo baja de forma caótica e incontrolada.

## Propuesta

Sistema controlado de reducción:
- `cpu_time > 80% × 10ms` durante N ticks → reducir a 80Hz temporalmente
- Si persiste → 60Hz, priorizar solo héroes (Tier 0)
- Recovery automático cuando baja la carga
